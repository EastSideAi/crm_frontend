# AI вкл/выкл + хэндофф менеджеру — дизайн (бот ↔ CRM)

Решает: «бот сам отвечает, но клиент может позвать человека; бот продолжает отвечать, пока
менеджер не увидит и не подключится; в CRM это видно и переключается». Источник истины по
диалогам — БД бота (`eastside-bot`, таблицы `bot_*`). CRM читает/пишет через прокси бэкенда.

## Модель данных (миграция 008 в eastside-bot, на `bot_conversation_state`)
- `ai_enabled boolean default true` — ведёт ли бот диалог автоматически. **false → бот НЕ
  отвечает сам** (входящие только сохраняются), диалог за менеджером.
- `handoff_requested boolean default false` — клиент попросил человека. **Бот при этом
  продолжает отвечать** (ai_enabled остаётся true) — просто флаг + уведомление в CRM.
- `handoff_reason text` — почему (что распознал тул / ключевое слово).
- `handoff_at timestamptz` — когда попросили.
- `taken_by text` — кто из менеджеров взял (имя из CRM).
- `crm_seen_at timestamptz` — когда менеджер последний раз открывал диалог в CRM (для unread).
- (`bot_user` += `channel text default 'telegram'` — под мультиканал; пока telegram.)

## Что меняет «тул» бота (добавим позже)
**Решает САМ бот** (не по ключевым словам — это убрано). Будущий инструмент `call_manager(reason)`
делает ОДНУ вещь — пишет в таблицу: `UPDATE bot_conversation_state SET handoff_requested=true,
handoff_reason=$reason, handoff_at=now() WHERE user_id=$id`. **Бот не замолкает** — отвечает
клиенту в духе «позвал менеджера, скоро подключится» и продолжает вести диалог по базе знаний.
Связь с CRM — через эту же таблицу (CRM читает её прокси-эндпоинтом). Пока тула нет — флаг никто
не ставит (в демо CRM показывает примерные хэндоффы для вида интерфейса).

## Что происходит в CRM после handoff_requested=true (полный цикл — РЕАЛИЗОВАНО)
1. CRM **поллит** `/admin/api/bot/conversations` раз в минуту (фоном, всегда).
2. У пункта меню **«Диалоги» загорается красный бейдж** с числом «просят менеджера» (видно с
   любого экрана, не открывая инбокс). На мобиле — бейдж на таб-баре.
3. В инбоксе такие диалоги **краснеют и всплывают наверх** (сорт: хэндофф → непрочитанные → время),
   на строке плашка «просит менеджера».
4. Менеджер открывает диалог → видит баннер «клиент просит менеджера» + историю; открытие шлёт
   `seen` (снимает «непрочитано»).
5. Жмёт **«Взять диалог»** → POST `/ai {enabled:false}` → `ai_enabled=false`, `taken_by=<имя>`,
   `handoff_requested=false` → бот перестаёт авто-отвечать (гейт в telegram webhook). Бейдж гаснет.
6. Менеджер отвечает клиенту в самом канале (композер в CRM — отдельный шаг позже).
7. Закончил → **«Вернуть AI»** → `ai_enabled=true` → бот снова ведёт.
8. (Следующий шаг, опционально) внутренний бот-уведомитель пингует команду в Telegram «клиент X
   просит менеджера» — мгновенно, не дожидаясь минутного поллинга.

## Поток
1. Клиент пишет в Telegram → сохраняем сообщение.
2. Читаем `ai_enabled`. **Выключен** → не зовём RAG, не отвечаем (ждём менеджера), CRM видит
   новое сообщение как непрочитанное. **Включён** → бот отвечает (как сейчас).
3. Клиент просит человека (тул/ключевое слово) → `handoff_requested=true`, бот продолжает.
4. В CRM: диалог всплывает наверх, красная плашка «просит менеджера»; менеджер видит.
5. Менеджер в CRM жмёт **«Взять диалог»** → `ai_enabled=false`, `taken_by=<имя>`,
   `handoff_requested=false`. Бот замолкает. (Ответ менеджера — в самом канале; композер в CRM
   позже.) Кнопка **«Вернуть AI»** → `ai_enabled=true`.
6. Открытие диалога в CRM → `crm_seen_at=now()` (снимает «непрочитано»).

## API (eastside-bot, новый router `/crm`, авторизация заголовком `X-CRM-Key` == CRM_API_KEY)
- `GET  /crm/conversations` — список: user_id, channel, имя/username, last_text, last_at,
  last_role, unread (last_at>crm_seen_at и role=user), ai_enabled, handoff_requested, taken_by, msgs.
- `GET  /crm/conversations/{user_id}/messages` — вся переписка.
- `POST /crm/conversations/{user_id}/ai` {enabled, by} — переключить AI.
- `POST /crm/conversations/{user_id}/seen` — пометить прочитанным (crm_seen_at=now()).
- `GET  /crm/analytics` — каналы, воронка, частые пробелы (kb_question_gap), счётчики, расход*.
  (*расход AI: если бот не логирует токены/стоимость — считаем приблизительно по числу
  ответов; точный — когда заведём usage-лог. Для MVP CRM показывает оценку.)

## Связь CRM ↔ бот (прокси через бэкенд, без секретов на фронте)
CRM-фронт (auth через crm_users `?k=`) → `eastside-backend /admin/api/bot/*` (require_user) →
HTTP к боту `BOT_API_BASE` с заголовком `X-CRM-Key=CRM_API_KEY`. Так фронт не держит секрет, а
бот не знает про crm_users. Бот и бэк — разные Supabase, поэтому связь по HTTP, не по БД.

## Что нужно от владельца (env + деплой), чтобы ожило
1. На **eastside-bot** (Railway): env `CRM_API_KEY=<любой длинный секрет>`; задеплоить (миграция
   008 применится сама на старте, если в _auto_migrate; иначе — Supabase SQL Editor).
2. На **eastside-backend** (Railway): env `BOT_API_BASE=https://<бот>.up.railway.app` и
   `CRM_API_KEY=<тот же секрет>`.
3. До этого CRM-инбокс показывает демо-данные (graceful fallback) — не падает.

## CRM-интерфейс (Telegram-like)
- Инбокс: сортировка по последнему сообщению, **непрочитанные** жирным + точка/счётчик, аватар
  канала, превью, статус (AI ведёт / просит менеджера / взят менеджером), фильтры каналов.
- Диалог: чат-бабблы, тумблер «AI ведёт», при handoff — баннер «клиент просит менеджера» +
  «Взять диалог»/«Вернуть AI». Открытие → seen.
